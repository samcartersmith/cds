Adopting CDS into a project can be a lengthy process, and should be done incrementally in specific phases. This guide will outline these phases and the appropriate steps to take.

Before we start, be sure that CDS has been installed into your project. Refer to the [Getting Started](../introduction/getting-started) documentation for more information.

## Preface

Some examples below are demonstrating manual usage of specific patterns, but if an applicable component currently exists in CDS, we highly encourage using it instead of writing something custom!

This ensures a high-quality product in the long run.

## Phase 1

### Color

The backbone of the CDS theme is our [color spectrum](../foundation/color). It's a critical piece as it ensures a consistent design across the board and unlocks advanced features like dark mode, and of course without it, all components would render without styles!

#### Design tokens

For web based projects, we support design tokens that can be interpolated into Linaria CSS or styled components. Tokens can be imported from the `@cbhq/cds-web/tokens` file and `@cbhq/cds-common/tokens/*` files.

```ts
import { css } from 'linaria';
import { styled } from 'linaria/react';
import { palette, spacing } from '@cbhq/cds-web/tokens';

const box = css`
  padding: ${spacing[1]};
`;

const Button = styled.button`
  background-color: ${palette.background};
`;
```

#### Palette hooks

When styling elements from within a React component, we encourage using the `usePalette` hook. This pairs perfectly with custom CSS variables (web only). With this being said, opt to use a CDS component that serves the same purpose instead of writing this kind of code manually.

<Tabs
  groupId="platform"
  defaultValue="web"
  values={[
    { label: 'Web', value: 'web' },
    { label: 'Mobile', value: 'mobile' },
  ]}
>
  <TabItem value="web">

```tsx
import { usePalette } from '@cbhq/cds-web/hooks/usePalette';

function Button() {
  const palette = usePalette();

  return <button style={{ '--button-bg-color': palette.background }} />;
}
```

  </TabItem>
  <TabItem value="mobile">

```tsx
import { View } from 'react-native';
import { usePalette } from '@cbhq/cds-mobile/hooks/usePalette';

function Button() {
  const palette = usePalette();

  return <View style={{ backgroundColor: palette.background }} />;
}
```

  </TabItem>
</Tabs>

## Phase 2

Once color and the `ThemeProvider` has landed, both typography and spacing can be done in parallel.

### Typography

[Typography](../foundation/typography) is a complex beast, so we discourage the use of custom text and heading styles. Because of this, you _must_ use the CDS provided [typography components](../components/text), as they've been thoroughly tested and provide a robust API.

<Tabs
  groupId="platform"
  defaultValue="web"
  values={[
    { label: 'Web', value: 'web' },
    { label: 'Mobile', value: 'mobile' },
  ]}
>
  <TabItem value="web">

```tsx
import { TextTitle1 } from '@cbhq/cds-web/typography/TextTitle1';

<TextTitle1>Prices</TextTitle1>;
```

  </TabItem>
  <TabItem value="mobile">

```tsx
import { TextTitle1 } from '@cbhq/cds-mobile/typography/TextTitle1';

<TextTitle1>Prices</TextTitle1>;
```

  </TabItem>
</Tabs>

#### Mobile assets

If integrating into a React Native project, the CDS fonts will need to be configured in your `react-native.config.js` file.

```js
module.exports = {
  assets: ['./node_modules/@cbhq/cds-mobile/icons/font', './node_modules/@cbhq/cds-fonts/native'],
};
```

### Spacing

If color is the backbone, then spacing is the muscle. It ensures the spacing within and around elements is consistent by aligning on 11 fixed unit sizes. A single spacing unit equates to 8px on normal scale, and 4px on dense scale.

#### Design tokens

Spacing tokens are also supported. Please refer to the [examples in color above](#design-tokens).

#### Spacing hooks

The `useSpacingStyles` hook can be used to provide inner padding on one, many, or any combination of sides. On web this returns a class name, while on mobile it returns a style object.

<Tabs
  groupId="platform"
  defaultValue="web"
  values={[
    { label: 'Web', value: 'web' },
    { label: 'Mobile', value: 'mobile' },
  ]}
>
  <TabItem value="web">

```tsx
import { useSpacingStyles } from '@cbhq/cds-web/hooks/useSpacingStyles';

function Row() {
  const spacingStyles = useSpacingStyles({ spacingTop: 1, spacingStart: 2 });

  return <div className={spacingStyles} />;
}
```

  </TabItem>
  <TabItem value="mobile">

```tsx
import { View } from 'react-native';
import { useSpacingStyles } from '@cbhq/cds-web/hooks/useSpacingStyles';

function Row() {
  const spacingStyles = useSpacingStyles({ spacingTop: 1, spacingStart: 2 });

  return <View style={spacingStyles} />;
}
```

  </TabItem>
</Tabs>

#### Offset hooks

CDS _does not_ support margins on a component as it's the parent's job to position its children accordingly. However, we do support negative margin through a concept known as offset. This allows children to expand outside of the parent container for instances where they need to expand to the edges of the screen/container (should be used rarely and carefully).

The `useOffsetStyles` hook can be used to provide outer _negative_ margin on one, many, or any combination of sides. On web this returns a class name, while on mobile it returns a style object.

<Tabs
  groupId="platform"
  defaultValue="web"
  values={[
    { label: 'Web', value: 'web' },
    { label: 'Mobile', value: 'mobile' },
  ]}
>
  <TabItem value="web">

```tsx
import { useOffsetStyles } from '@cbhq/cds-web/hooks/useOffsetStyles';

function Row() {
  const offsetStyles = useOffsetStyles({ offsetHorizontal: 1 });

  return <div className={offsetStyles} />;
}
```

  </TabItem>
  <TabItem value="mobile">

```tsx
import { View } from 'react-native';
import { useOffsetStyles } from '@cbhq/cds-web/hooks/useOffsetStyles';

function Row() {
  const offsetStyles = useOffsetStyles({ offsetHorizontal: 1 });

  return <View style={offsetStyles} />;
}
```

  </TabItem>
</Tabs>

## Phase 3

### Components

Once the core foundational pieces have been integrated, we suggest using all the components currently available in CDS. We suggest migrating a codebase 1 component at a time, to ensure proper coverage. The following order is suggested, but not mandatory.

- Typography (should be done)
- Icons
- Buttons
- Links
- Box, card, etc
