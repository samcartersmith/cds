Alert dialogs are specific types of modal overlay that communicate critical information and block the user from completing a task until a specific action is taken.
Alerts are highly disruptive and should be used sparingly, especially with high-risk actions that might have negative or destructive consequences.
Alert dialogs are different but related to other types of user feedback and communication patterns like Banners, Callouts, Inline Validation, and Trays.

## Usage

**Critical error**
Something bad happened, requires user action to proceed

**Warning**
High importance information requiring user acceptance

**Confirmation**
Yes/No confirmation before destructive or consequential action

**Alert Action Button Layout**

- Generally, use two-button alerts. Two-button alerts provide an easy choice between two alternatives. Single-button alerts inform, but give no control over the situation. Alerts with three or more buttons create complexity, which is a bad user experience.
- Include a Cancel button that provides a clear, safe way out of a dialog and returns the computer to the state it was in before the dialog appeared. The Esc (Escape) key is mapped to the Cancel button.
- For the two button option, try to fit the label in the “Side-by-Side” variant. If the label doesn’t fit, use “Stacked”.

## Types

### Error

<Image srcLight="/img/error/alert_light.png" srcDark="/img/error/alert_dark.png" />

Human or system errors prevent the app from working or the user from continuing.

Alert errors convey important or urgent information related to the state of the app or the device. A potential blocker in the flow (user cannot progress any further).

Use alert errors sparingly. Alerts disrupt the user experience and should only be used in important situations like notifying people about problems. The infrequency of alerts helps ensure that people take them seriously.

### Warning

<Image srcLight="/img/warning/alert_light.png" srcDark="/img/warning/alert_dark.png" />

Warning alerts create friction in a user flow in order to confirm the user is aware of the consequences of their action. They are often triggered because these consequences could be unwanted or destructive and we want to be sure the action was not taken accidentally. Warning alerts allow for the user to continue as originally intended but buttons often emphasize what we perceive as the least destructive action.

## Content

Alerts are meant to convey information quickly and effectively. Use the title to succinctly state the problem or request. If appropriate, offer a solution in the body copy.

- Use the title to state the problem or request clearly and succinctly
- Offer solutions or explanations in the body copy, if appropriate
- Don’t repeat the info in the title again in the body copy
- Avoid apologizing
- Keep titles to 1-2 lines and body copy to 1-3 lines
- Because button space is tight, use concise 1-2 word labels

For more guidelines on writing alerts, including examples, check out [Alerts and Errors](https://docs.google.com/document/d/1QnxOOjhqUb8ENrGt3hQTTDCDxnCPRyEfkO-U-ftm4uc/edit#heading=h.bqjlx0cnlsjn) in the content design guide.

## Behaviour

### General

- When appearing, the alert modal moves down 16pt and fades from 0 to 100 opacity.
- When disappearing, it simply fades out.
- An alert can only be dismissed using the provided actions buttons or the escape key. The escape key It can’t be dismissed with an “x” or by clicking outside of the modal.
