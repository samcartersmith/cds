## Responsive

CDS layout components support responsive style properties. You can pass a configuration object to any layout component and specify unique styles for phones, tablets, or desktops.

<DocImage category="foundation" component="layout" name="responsive" format="png" />

```jsx
// hoisted configuration object
const responsiveConfig = {
  phone: {
    spacing: 2,
  },
  tablet: {
    spacing: 3,
  },
  desktop: {
    spacing: 4,
  },
};

// in the render method
<Card responsiveConfig={responsiveConfig} />;
```

### Supported Properties

Box, HStack, VStack, Card, and Typography:

- Spacing
- Offset spacing
- Gap
- Flex styles (justify content, align content, align items, align self, flex direction, and flex wrap)
- Display
- Visibility

ListCell, TableCell, TableRow, and ContentCell:

- innerSpacing
- outerSpacing

### How It Works

Under the hood, your responsive styles are wrapped in a media query that corresponds to the specified device:

- **Phone**: `max-width: 767px`
- **Tablet**: `max-width: 1279px`
- **Desktop**: `min-width: 1280px`

If you create a configuration with only one or two devices, the missing device will use any default styles, eg: If you have an HStack with a responsive config like `phone: { spacing: 4 }, tablet: { spacing: 2 }` and `spacing={1}` desktop styles will default to `spacing={1}`. The only exception is if you do not specify a phone style, it will use the tablet styles (if specified) since the tablet styles use max-width.

### SSR & SSG Support

Responsive styles are SSR (Server Side Rendering) and SSG (Static Site Generation) friendly. Styles use pure CSS with no conditional Javascript render, so your components will use responsive styles on first paint. That means no more server/DOM mismatches or flickering after first paint as the DOM rehydrates to match the device specified styles.

### Performance

You should always hoist or memoize your responsive style configurations. There are notable performance degradations if you leave your responsive style configurations inside your components and fail to memoize them, because they are recreated on each render.

## Breakpoint Hooks

In some cases, you may want to conditionally render a component or tree of components. In that case, we recommend that you use the `useBreakpoints` hook.

See [useBreakpoints documentation](/hooks/useBreakpoints) for usage.

CDS does not recommend using `useBreakpoints` for responsive styles, instead we strongly advise that you use the responsive style props as they use native CSS instead of Javascript to render styles. We recommend that you only use `useBreakpoints` to conditionally render components that make network calls or large components trees.

You should only use `useBreakpoints` as a last resort. For example, if a style prop is not supported by responsive styles you should use `useBreakpoints`. Make sure you always specify default styles for each responsive style prop which will be used for initial hydration.

### Risks of Using Javascript for Responsive Styles

Using Javascript for responsive styles can lead to flashes on initial hydration in SSR and SSG environments: the default styles will render on first paint until the screen width can be measured, and then the device-specific styles will render.

`useBreakpoints` can also lead to cumulative layout shifts (CLS) if the default styles differ from a user’s device and there are responsive styles specified for that device, eg: the default styles are for desktops, after initial paint the window will be measured and that device’s specified styles will be rendered causing the layout to shift suddenly.

You can prevent this by setting a default device, which will initialize the `useBreakpoints` hook using Context. Or by using the [BrowserOnly](/components/browser-only) higher order component or the [useIsBrowser](/hooks/useIsBrowser) hook to wait for the component to mount before rendering any content. This ensures that the window object is present in the DOM, which is necessary to determine the user’s device.
