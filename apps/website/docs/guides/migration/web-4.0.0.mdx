## [4.0.0]

### Icon and NavigationIcon

We've changed how [Icon](/components/icon/) and [NavigationIcon](/components/navigation-icon/) handle accessibility to better align with our current standards.

<TextTitle4 as="h4">Removed Props (Icon and NavigationIcon)</TextTitle4>

```ts
/**
 * Give the icon a description. It will show up in a tooltip after
 * hovering over the icon for 3s. This is a default browser feature and
 * very helpful to users to understand what the icon does.
 */
title?: string;
/**
 * If the icon is labeled by an independent element, then provide that element's id.
 * This is used to set the aria-labelledby value on the icon. If the `title` prop is
 * also provided, then `labeledBy` will be used to set the id on the `<title>` element.
 */
labeledBy?: string;
```

<TextTitle4 as="h4">Removed Props (NavigationIcon)</TextTitle4>

```ts
/** Accessibility hint for assisted devices when focused.
 * Message should indicate the purpose of the control.
 * If not supplied, will default to label */
accessibilityHint?: string;
/**
 * Accessibility prop that describes the component that labels it by its `id`
 * @link https://developer.mozilla.org/en-US/docs/Web/Accessibility/ARIA/ARIA_Techniques/Using_the_aria-labelledby_attribute
 */
accessibilityLabelledBy?: string;
/** Used for a11y to indicate the relationship between a controlled element and its controller
 * eg: a Button that controls the visibility of a Modal
 */
id?: string;
/** Used to associate a label with an input for accessibility */
accessibilityLabelId?: string;
/** Used to associate descriptive text to an input for accessibility */
accessibilityDescriptionId?: string;
```

<TextTitle4 as="h4">New Props (Icon)</TextTitle4>

```ts
/**
 * Accessibility label describing the control.
 * If no accessibilityLabel, it will set it to label
 * Maps to `aria-label`
 */
accessibilityLabel?: string;
```

<TextTitle4 as="h4">Migration Guidance</TextTitle4>

If you're using `title` and `labeledBy` to set an accessible label for your stand-alone icon, use `accessibilityLabel` instead. Internally, we mirror the icon's `title` attribute to `accessibilityLabel` to retain the browser's native tooltip functionality:

```diff
  <Icon
    name="add"
    size="m"
-   title="Plus icon"
-   labeledBy="add-icon-id"
+   accessibilityLabel="Plus icon"
  />
```

If you're using `labeledBy` to reference the id of another text element as an accessible label, this is no longer supported. We're not maintaining support for this use case because any visible label that is presented with the icon should already provide the necessary context, and the icon should remain purely decorative. However, if the icon is referencing an invisible text element, use `accessibilityLabel` to directly set the label instead:

```diff
- <TextBody id="plus-icon-label" as="span" dangerouslySetClassName="visuallyHidden">
-   Plus icon
- </TextBody>

  <Icon
    name="add"
    size="m"
-   labeledBy="plus-icon-label"
+   accessibilityLabel="Plus icon"
  />
```

### IconButton and NavigationIconButton

If you employ A11Y validation, you may notice that your [IconButtons](https://cds.cbhq.net/components/icon-button/) and [NavigationIconButtons](https://cds.cbhq.net/components/navigation-icon-button/) start to fail validation checks:

```
Error: Buttons must have discernible text
```

This occurs because icons are now `aria-hidden` by default and no longer count as valid content when deriving an accessible label. So, if your icon buttons aren't properly labeled, A11Y validation tools will surface errors. However, these are easily resolved by supplying accessible labels to your icon buttons:

```diff
  <IconButton
    name="add"
    variant="primary"
    onPress={handlePress}
+   accessibilityLabel="Add an item"
  />
```
