This guide provides a development path for common use cases that CDS developers come across. Sections marked with <UnknownIcon />
represent topics where a final decision has not been made. Please check with the CDS team before using an undecided convention.

This guide is specific to CDS components and is not intended to be a general purpose React / Typescript guide. See [Frontend Standards & Code Conventions](http://go/fe-standards) for more coding conventions.

## Prop Naming <UnknownIcon size="m" />

Generally speaking props should clearly describe the functionality of a component and what the component does. Props should not describe why a component does something.
For example `isMobileScreen` describes the browser window size, and the reason for variation, but not the component itself. A better name could be `isCompactLayout`.

The following conventions can be used for props.

### boolean <UnknownIcon />

Use a prefix that clearly indicates a boolean value `is` `can` `has` `should`

**Examples**

```jsx
<Component isVisible={false}>
  <div />
</Component>
```

```jsx
<Section shouldShowSubSection={true}>
  <div />
</Section>
```

```jsx
<Element isCompact={true}>
  <div />
</Element>
```

### numeric <UnknownIcon />

Use a prefix (num) or postfix (count, index) that can imply a number. `numItems` `itemCount` `itemIndex` `numValues`

**Example**

```jsx
<Element numItems={300} />
```

### array <UnknownIcon />

Use plural nouns `items` `elements` `nodes` `children` `rows` `columns`

**Example**

```jsx
<Element rows={rows} />
```

### object <UnknownIcon />

Use a noun to denote an Object. `item` `user` `person` `location`

**Example**

```jsx
<profile user={user} />
```

### node <UnknownIcon />

Use prefix node or nodes `containerNode` `childNodes`

### element <UnknownIcon />

Use prefix element or elements `hoverElement` `listItemElements`

## Type naming <UnknownIcon size="m" />

### Union Types <UnknownIcon />

Use singular naming

```tsx
type Position = 'relative' | 'absolute' | 'fixed';
```

### Prop Types <UnknownIcon />

Add a suffix of `Props`

```tsx
type ButtonProps = ButtonBaseProps & PressableProps;
```

## Overlay components naming

Overlay components provides an additional layer on top of current interface.
They can be toggled on/off using a boolean prop.
Current CDS Overlay components:

- [Modal](/components/modal)
- [Alert](/components/alert)
- Overlay
- Tray
- Select
- Drawer
- Toast
- Tooltip
- Popover

All the overlay components above should follow the following conventions to be consistent.

### Visibility

Overlay components visibility should be controlled using the `visible` prop.

#### Controlled :heavy_check_mark:

```jsx
const [visible, { toggleOn, toggleOff }] = useToggler(false);

<Modal visible={visible}>{children}</Modal>;
```

#### Uncontrolled :x:

```jsx
const [visible, { toggleOn, toggleOff }] = useToggler(false);

{
  visible && <Modal>{children}</Modal>;
}
```

### Visibility Handling

Methods that control component visibility should have declarative names that mirror
the indicate the desired behavior. For example, visibility handlers for a
`PopoverMenu` might be named:

- `openMenu`
- `closeMenu`

```jsx
const [visible, { toggleOn, toggleOff }] = useToggler(false);

{
  visible && (
    <PopoverMenu openMenu={toggleOn} closeMenu={toggleOff}>
      {children}
    </PopoverMenu>
  );
}
```

#### Accessibility

If a component's visibility is controlled by another component, use the
`useAccessibleControlledVisibility` hook which will add accessibility props to
both the trigger and the controlled component.

```jsx
const accessibilityLabel = 'open-menu-button';
const [visible, { toggleOn, toggleOff }] = useToggler(false);
const { triggerAccessibilityProps, controlledElementAccessibilityProps } =
  useAccessibleControlledVisibility(visible, accessibilityLabel);

<Trigger {...triggerAccessibilityProps}>Click me!</Trigger>;
{
  visible && (
    <PopoverMenu {...controlledElementAccessibilityProps}>My visibility is controlled</PopoverMenu>
  );
}
```

### Lifecycle methods naming

Lifecycle methods are used to operate on component visibility state changes.
Provide the lifecycle methods that you feel are likely needed, and add more methods base on consumer requests.

| Method     | Description                                    |
| ---------- | ---------------------------------------------- |
| onWillShow | Callback fired before the component shows.     |
| onDidShow  | Callback fired after the component is visible. |
| onWillHide | Callback fired before the component hides.     |
| onDidHide  | Callback fired after the component is hidden.  |
