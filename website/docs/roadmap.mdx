### Unification

#### Unify our primitives (mobile and web), which will better position us and cut future revision and support cost

Primitives such as Typography, Color, Layout are the most frequently used shared resources within a Design System. By offering these holistically, we immediately provide significant value to feature teams. We also prove that the unification is happening. Last half CDS invested in expanding Typography and Color to support more needs such as Legibility, Scale, and an algorithmic color approach.

- ✅ Text component should assign appropriate styles internally based on active scale.
- ✅ Text component color prop should assign appropriate styles based on active palette.
- ✅ On web, update Text component to include a required `as` prop. Typescript types extend based on element passed in.
- Icon component color prop should assign appropriate styles based on active palette.
- On mobile, migrate Button to pull colors from palette.
- _The following are contingent on ongoing design discussions_
  - Align on heavy/light usage in Icon component.
  - Unify Icon component API between web and mobile based on heavy/light outcome.
  - Create an asset icons package which is dark mode compatible.
  - Create an illustrations package which is dark mode compatible.

### Component port

Mobile has a wealth of good quality components we can port to the web. For each component, we will audit it’s compatibility for the Web, and make simple alterations needed for mobile (i.e. api naming, etc). We will prioritize components that are quick and seamless to transition to the web. This may result in some components that don’t replace all their similar components, but it will replace a significant number of one-off components.

- 🚧 On web, update Text component to use new API based on Tailwind exploration work.
- Create Box primitive. Replace `View` on mobile with `Box`.
- Box component backgroundColor prop should assign appropriate styles based on active palette.
- Box component spacing prop should assign appropriate styles based on active scale.
- On web, create Flex and Grid components which extend Box primitive.
- _The following require more details based on design outcomes_
  - Buttons
  - Inputs
  - Controls
  - Cards
  - Sections
  - Elevation
  - Table/Cells

### Fill in the gaps

There are components on mobile i.e. Tab Bars or Trays that may not make sense as-is on the Web, so we’ll look at how we can unify the heuristics, naming, and guidance, but provide web-specific equivalents.

- Continue audit & port components from React Native to Web that are higher cost and lower relevance

## Goals

- **Primitive Adoption**: To build a sense of consistency within Coinbase experiences, it’s important for us to have a high percentage or complete adoption of CDS primitives across all experiences. This will cut down variation and bugs while also streamlining contribution.
- **Comprehensive web & mobile component library**: We want to ensure that web and mobile teams have the primitives, components, and resources to take on web and mobile work, so that they’re not starting from scratch and are building consistent experiences. While there is overlap between mobile and web, there may be some components that are web or mobile specific, and that’s ok.
- **Partner**: In order for us to be effective in our work, we need to actively work on relationships and bring teams along with our work. We need to seek out opportunities to collaborate with other teams, facilitate work between teams, and be a point of contact for any new pattern.
